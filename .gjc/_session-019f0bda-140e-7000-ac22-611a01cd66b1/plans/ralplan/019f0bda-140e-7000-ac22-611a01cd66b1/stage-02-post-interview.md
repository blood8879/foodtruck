# Post-ralplan Intent Reconciliation (자동 모드)

> consensus: Critic2 OKAY. 아래는 루프가 가정으로 해결했거나 명세·의도와 충돌 가능한 항목. 승인 게이트에서 사용자 확인 필요.

## 미확인 가정 (승인 전 확인)
1. **수익화는 원 명세에 없던 신규 추가**: deep-interview 명세에는 수익화가 없었고, 사용자가 계획 중 추가. 계획은 MVP(M1~M3)에 entitlement seam+영업세션+광고 슬롯 위치만 넣고, 실제 광고 SDK·구독 결제는 M4로 분리. → 이 분리(MVP는 설계만, 실제 과금 M4)가 의도와 맞는가?
2. **동기화 기본 BUY**: A-1(PowerSync/Electric/Couchbase 등) 권장 — 벤더 종속·월 비용 발생. 커스텀(A-2)은 H1/M-OBX/H3 골든 통과 시에만 조건부. → 벤더 도입·비용 OK인가, 아니면 무종속 커스텀 선호?
3. **기술스택 B-1 (RN/Expo + React + 공유 TS 코어 + Postgres/Node)**: 사용자가 명시 선택한 적 없는 기술 가정. → 이 스택 수용 가능한가?
4. **무료/유료 경계 (사업 결정)**: 계획 제안 = 무료(트럭1·메뉴 N개·당일매출) / 유료(다중직원·기간분석·그래프·PC웹·내보내기). 가정이므로 사업 관점 확인 필요.

## 명세 충돌 (해소 필요)
5. **[HIGH] 다중 직원 게이팅 ↔ 핵심 시나리오 충돌**: deep-interview는 '사장이 직원 추가, 사장·알바가 여러 기기에서 동시 입력'을 핵심 요구로 확정(R5b,R6)했다. 그런데 계획의 무료/유료 경계에서 '다중 직원'을 유료로 게이팅하면, 무료 MVP 사용자는 정작 핵심 시나리오(알바와 동시 주문)를 못 쓴다. → 해소 옵션: (a)다중직원을 무료에 포함하고 다른 것(기간분석·PC웹·내보내기)을 유료로, (b)무료는 직원 N명까지 허용 후 초과부터 유료. 사용자 결정 필요.
6. **[LOW] M4 구독 결제 ≠ 명세의 '외부 결제 연동(향후)'**: 명세 Non-Goal '외부 결제'는 푸드트럭 손님 결제(현금/카드)를 뜻했고, M4 구독 결제는 앱 구독료(사장→운영사)로 별개. 충돌 아님 — 정보성 명시.
